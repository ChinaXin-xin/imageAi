# AGENTS.md

## 项目结构

- `pom.xml`：Spring Boot 3.5.14 后端工程配置，JDK 17。
- `src/main/java/xin/students/imageaioriginal/`：后端 Java 源码入口。
- `src/main/resources/application.yaml`：后端应用配置，包含 CLI Proxy 管理端地址和管理密钥。
- `src/test/java/xin/students/imageaioriginal/`：后端测试。
- `vue3/`：Vue 3 前端代码，本项目所有 Vue 代码统一放在这里。
- `.mvn/`、`mvnw`、`mvnw.cmd`：Maven Wrapper。

## 技术栈

- 后端：JDK 17、Spring Boot 3.5.14、MyBatis-Plus、MySQL 8、Redis。
- 前端：Vue 3、Vite、TypeScript、Element Plus。
- 额度接口来源：Cli-Proxy-API-Management-Center / CLI Proxy API Management API。
- 前端不要直接访问管理端，统一调用本项目后端 `/api/**` 接口。
- 管理 API：`/v0/management`，通过 `Authorization: Bearer <管理密钥>` 鉴权。

## 运行命令

后端：

```bash
./mvnw spring-boot:run
```

Windows：

```bash
mvnw.cmd spring-boot:run
```

前端：

```bash
cd vue3
npm install
npm run dev
npm run build
```

前端默认通过 Vite 代理把 `/api` 转发到 Spring Boot：

```bash
cd vue3
npm run dev
```

## 开发规范

- 不新建无关项目，所有前端代码放在 `vue3/` 下。
- 不在前端代码中硬编码管理密钥、API Key、Token 等敏感信息。
- 管理端访问由后端封装，前端只访问本项目后端接口。
- 前端接口访问统一封装在 `vue3/src/services/`。
- 页面优先使用 Element Plus 组件，保持加载态、错误态、空态完整。
- 添加生图任务时机型为必填项，前端必须阻止空机型提交，后端也必须兜底拒绝空机型。
- 后端和前端改动保持小范围，不做无关业务和大规模重构。
- 后端业务持久化代码使用 MyBatis-Plus Entity/Mapper/Wrapper；建表初始化和补列可保留轻量 DDL 兜底，但业务 CRUD 不再在 Service 中手写 JDBC。
- `ImageTaskQueueService.java` 只保留任务编排逻辑，单文件不得超过 1000 行；提示词拼接、文件处理、下载打包、MyBatis-Plus 持久化等能力拆到独立小文件。
- 新增配置示例使用 `.env.example`，真实 `.env.local` 只保留在本地。
- 每次新增、修改、删除或查询项目长期约定时，都要同步维护 `AGENTS.md`。
- 最终生图提示词按“真实产品结构锁定/参考图优先级 → 上传图深析结果 → 手机膜/镜头膜结构生成规则 → 套装规格数量锁定 → 任务参数/规格 → 主图/介绍图画面要求或排版图专用填充生成要求 → 排版图直接参考约束/参考风格图风格 → 负面约束/生成要求”的顺序拼接；不要把深析上传图的提示词本身拼进最终生图提示词，也不要把排版图解析结果拼进最终生图提示词。
- 手机膜、镜头膜、保护壳等精密配件生图必须先锁外轮廓、孔位数量、孔位位置、孔位大小差异和非对称结构，再应用参考风格图和风格词；风格词不能覆盖上传实拍图结构。
- 镜头膜和手机膜支持多品牌、多型号，不能把任意单一型号的孔位结构写死成通用规则；必须先根据上传图和深析结果智能识别当前机型，再锁定对应外轮廓、孔位数量、位置、大小差异和一体式/分离式形态。
- 调用实拍图深析接口时，必须把用户输入的机型作为提示词最前方内容拼接后再提交给 GPT 分析。
- 最终生图提示词要保持短而明确，深析结果和参考风格图风格进入生图前需要做长度控制，避免多参考图任务因提示词过长导致生图接口超时。
- TEMU/Amazon 主图按平台首图标准生成：不要文字、图标、角标、贴纸文案和水印；主图差异化优先通过左右位置、俯拍/斜拍角度、手机颜色、背景光影、轮廓光、清晰度和轻微 3D 层次实现，避免重复铺货感。
- 当主图或介绍图生成数量大于 1 且当前类型未启用排版图时，后端需先调用 GPT 基于对应最终提示词规划 JSON 场景数组，再把每个场景描述拼入对应图片的生图提示词，确保多张图场景、构图、背景、光影或卖点表达不同；当前类型启用排版图时，不再拼接原有“本张图片场景规划”，改为拼接“基于排版图填充上传素材并生成电商图”的专用提示词。
- 多张主图必须显著裂变：至少通过钢化膜/屏幕膜在手机左侧或右侧的摆放变化、俯拍/斜拍角度变化、轮廓光/背景光影变化来区分；任务未指定手机颜色时可变化手机颜色，但不得改变机型和产品结构。
- 手机膜生图必须以手机机身/屏幕作为比例尺：手机主体完整入画并保留安全边距；屏幕膜/钢化膜/防窥膜必须与手机屏幕长宽比和覆盖尺寸匹配；镜头膜只能匹配后摄区域，不能放大到接近半个手机；清洁/安装配件按参考图等比例缩放并整齐摆放。
- 启用排版图时，排版图是版式骨架约束，优先级高于场景规划和参考风格图，仅低于实拍图产品结构；场景规划只能在排版图主体区、配件区、信息区、留白、安全边距、网格/分栏和对齐关系内调整光影、角度、背景或层次，不能重新散乱摆放。
- 当前图片类型启用排版图后，不再拼接原有的主图或介绍图画面要求，改为拼接“基于排版图填充上传素材并生成电商图”的专用提示词；必须严格遵循排版图的画面结构、主体位置、配件位置、留白比例和构图关系，将上传产品图、配件图等素材填入对应区域。排版图与原始画面排版要求冲突时，以排版图为最高版式优先级，但不得覆盖实拍图产品结构、真实比例和套装数量。
- 当前图片类型启用排版图并调用生图接口时，后端必须把排版图作为该类型参考图列表的最后一张输入，并在每张图片提示词里明确“排版图/样板图是最后一张输入图片”，要求模型把前面输入的产品图、配件图、壁纸图等素材填充到最后一张排版图对应区域。
- 多场景规划结果进入生图前需要做自审查：如果场景描述包含额外包装、包装盒、小黑包、收纳袋、卡片、托盘等未选择物品，应丢弃该场景并使用安全备用场景。
- 每张最终生图提示词拼入场景规划后，还需追加成品级自审与修正约束：只允许白名单物品，若场景、参考风格图风格或模型联想引入包装、收纳袋、托盘、支架、底座、展示道具或未选配件，必须忽略并禁止生成。
- 当前客户只上传或选择手机膜相关内容：手机、钢化膜、高清膜、防窥膜、镜头膜，以及已上传/已选择的清洁或安装辅助配件。除任务已选择或上传的这些物品外，不要生成包装盒、收纳袋、卡片、托盘、支架、底座、展示道具或其他赠品。
- 任务上传入口不再提供“包装图”和品牌图；前端和后端任务创建只保留实拍图、排版图、壁纸图以及已选择的额外配件；主图/介绍图参考风格图从参考风格图库中选择。
- 任务上传的排版图、壁纸图必须分别支持选择用于主图或介绍图，默认主图和介绍图都使用；排版图只用于图片填入版式/布局参考，不做 GPT 深析或版式解析，不等同于参考风格图；后端按用途将排版图和壁纸图作为生图参考图直接传入。
- 图片比例按主图和介绍图分别设置，主图默认 `1536:1536`，介绍图默认 `1536:1536`；如果某一类型选择使用排版图，该类型只能使用“自动比例”，后端按排版图实际宽高比计算该类型输出尺寸。
- 前端主图比例和介绍图比例放在同一行展示，位于平台选择下方；启用排版图的类型只展示“自动比例”。
- 前端额度页展示“可生成图数量”时，使用后端返回数量除以 `1.5` 后向下取整展示。
- 手动暂停任务的提示语使用紫色文字展示，不使用错误红色；真正失败错误仍使用红色。
- 排版图与参考风格图按图片类型互斥：如果任务上传了排版图并选择用于主图，则不能选择主图参考风格图；如果任务上传了排版图并选择用于介绍图，则不能选择介绍图参考风格图，前端需禁用并清空对应选择，后端需兜底忽略对应参考风格图 ID。
- 任务上传的壁纸图不做 GPT 深析或自动识别，生图时按已选用途作为参考图直接贴用；壁纸直接贴到手机屏幕或屏幕展示区域。
- 清洁/安装辅助配件不能写死形状、颜色、尺寸或文字；不同手机型号和套装的配件形态可能不同，必须按上传参考图和深析结果生成。若参考图配件上有文字、图案、标签或封边结构，最终图必须把这些作为配件外观结构保留；不要凭空加字、改字、漏字、生成无字替代品或把文字区域替换成空白块。
- 高清/钢化膜与防窥膜必须按任务选择严格区分：高清/钢化膜是透明、清亮、高透玻璃质感；防窥膜是深色、灰黑或轻微褐色防窥质感。未选择防窥时不要把高清膜生成为褐色/暗色防窥膜，未选择高清时也不要把防窥膜生成为普通透明膜。
- 任务详情不展示生图接口返回的 `revised_prompt`，避免和最终生图提示词混淆。
- 图片放大查看器要适配横屏笔记本：小图要放大到合适尺寸，大图不能超出可视区域；蒙版使用 `rgba(25, 25, 25, 0.88)`，点击非图片区域关闭。
- 用户界面中商品资料上传位统一称为“排版图”，用于图片填入版式/布局参考；“参考风格图”只指参考风格图库中的主图参考风格图和介绍图参考风格图。历史代码/API/数据库字段可继续沿用 `targetTemplate`、`template` 等内部命名以兼容旧数据。
- 参考风格图分为主图和介绍图两类，只用于对应类型的生成风格参考；主图参考风格图只能拼入主图生图提示词，介绍图参考风格图只能拼入介绍图生图提示词。
- 参考风格图图片仍需用于前置风格分析；生图时只按主图/介绍图类型拼接对应参考风格图风格文字，不再把参考风格图图片作为生图参考图传给模型；不得让参考风格图风格覆盖实拍产品结构、配件数量、孔位锁定和白名单约束。
- 最终生图提示词里不要展示参考风格图文件名，也不要写“参考风格图图片不传给模型”之类的实现说明，只保留对应类型的参考风格图风格分析和约束。
- 上传图片后的前端展示必须使用缩略图预览，不展示 Element Plus 默认文件名列表；长文本“查看全文”弹窗使用 Markdown 渲染。
- 任务详情里的提交图片卡片使用缩略图展示；点击放大和下载必须走原图接口，不能继续使用缩略图。
- 前端上传 AVIF 等非常规图片格式时，应优先在浏览器端转成 PNG 后再提交；后端分析图片时遇到 ImageIO 无法解码的图片，要保留原始 data URL 兜底，避免排版图或参考风格图上传直接 500。
- 生图接口需要携带上传原图或额外配件参考图时，后端先将参考图等比压缩为适合 API 传输的图片，避免原始大图导致请求超时。
- 生图接口响应不能假定一定是 `application/json`；如果返回图片二进制、`application/octet-stream` 或 base64，后端要保存为本地图片文件，数据库只保留本地路径/访问 URL，不再把生成图片 base64 长文本作为结果长期保存。
- 生图任务执行支持并发，需通过 `image-ai.image-generation.max-task-concurrency`、`max-images-per-task`、`max-global-image-concurrency` 控制任务并发、单任务图片并发和全局图片并发；默认单任务图片并发和全局图片 API 并发为 10，不要使用无限制线程池直接打满外部 API。
- 单张图片生成遇到超时、HTTP 5xx、连接中断、空响应等外部接口临时异常时，后端需自动重试 2 次；同一结果最多尝试 3 次，只有最终失败才写入失败状态。
- 暂停或删除任务时必须取消该任务正在进行的所有本地生图请求，并确保后续返回结果不会写入已暂停或已删除任务。
- “套装规格与卖点”中的套餐规格必须从数据库中的“额外配件”库选择，配件包含名称和图片；任务保存所选配件 ID、名称与数量，数量最低为 1。
- 额外配件用于套餐规格时，生图必须把已选择配件的原图作为参考图传入，并在最终提示词里要求按数量精确出现，未选择的配件不要出现。
- 参考风格图分析提示词必须可在前端默认设置里修改并保存到数据库，不能硬编码在参考风格图分析逻辑中。
- 用户界面统一称为“场景图提示词”；该提示词必须可在前端默认设置里修改并保存到数据库，添加任务页也必须提供任务级“场景图提示词”输入框并默认带入默认设置内容。
- 后端多场景规划和兜底场景必须优先使用任务级“场景图提示词”；任务级内容为空时才由后端根据最终提示词、卖点和套装规格自动规划，不得在代码中写死固定场景循环覆盖用户输入。
- 主图生成数量大于 1 时，第 1 张主图必须作为套装合集图，手机、膜片/镜头膜和所有已选择配件按真实比例同框整齐展示；其它主图或介绍图若未启用排版图，可按场景卖点从已选择配件中自动选择部分配件展示，不要求每张图都展示完整套装，但不得出现未选择配件。
- 添加任务提交时，如果商品名称为空或仍是系统随机生成风格的 8 位数字字母串，前端需重新生成新的随机商品名称后再提交。
